# Введение
  
Игровая индустрия является одной из самых динамично развивающихся отраслей IT. Оборот которой в 2025 году достигает рекордного значения в **197 млрд USD**, как утверждают [Newzoo](https://newzoo.com/resources/blog/year-in-review-2025-to-date). Однако, несмотря на обилие инструментов и информации, большинство проектов, начинаемых школьниками, студентами, не доходят до стадии рабочего прототипа. Это объясняется отсутствием понимания методологии Minimum Viable Product (MVP) – это минимальная версия, после которой продукт пригоден для эксплуатации, которая также является стандартом IT индустрии. Освоение этого метода позволяет не только создавать играбельные версии продуктов, но и формирует универсальный навык: умение превращать идею в проверяемую модель, а затем - в готовое решение.  
  
Таким образом, возникает противоречие: с одной стороны метод MVP является признанным стандартом в профессиональной среде, с другой начинающие разработчики не владеют им, как следствие, не могут довести свои идеи до работающего продукта.  
  
Теоретическую базу проекта составили работы Джесси Шелла «[The Art of Game Design](https://www.amazon.com/Art-Game-Design-Lenses-Second/dp/1466598646)» (2014), Тодда Варфела «[Prototyping: A Practitioner’s Guide](https://rosenfeldmedia.com/books/prototyping/)» (2018), а также видеокурсы российского разработчика [ITProgger](https://www.youtube.com/@ITProgger) по библиотеке PyGame. В данных источниках рассматриваются основы игрового дизайна, принципы прототипирования и практические аспекты разработки игр.
  
Цель: Показать процесс создания рабочего игрового прототипа со всеми запланированными функциями.  
  
Задачи:  
- провести сравнительный анализ существующих ПО для выбора инструментальных средств разработки игрового прототипа;  
  
- освоить необходимые навыки: язык программирования и библиотеки, необходимые для создания программы;  
  
- создать модель будущей игры;  
  
- реализовать главные механики прототипа;  
  
- протестировать и оценить степень готовности прототипа.  
  
Методы: анализ, синтез; сравнительный анализ; моделирование; практическое программирование (Python, PyGame); тестирование и отладка прототипа.

## **2.2 Архитектура проекта и загрузка уровня**  
  
Для разработки основной механики необходимо сделать загрузчик уровней. Для её реализации, нужно составить базовый макет уровня, содержащий все объекты. В роли макета я предпочту ASCII модель. Она хороша простотой реализации и высокой скоростью считывания. Для реализации загрузки, была объявлена функция, она открывает файл с уровнем, разбивает его построчно и добавляет в список. Перебирая его, мы получаем индекс символа и заносим его в свой список однотипных объетов. Это нужно для определения координат каждого объекта, они рассчитываются по формуле: 

- ```X = (_j_ + 1) × 64```,
- ```Y = (_i_ + 1) × 64```.

- `j` - индекс символа в массиве
- `i` - номер строки, в которой находится символ
- `64` - константа, длина каждого квадратного объекта

Прибавляем 1, чтобы объекты не прижимались к левому краю, так как нумерация в Python начинается с нуля. В итоге функция возвращает - данные уровня и координаты игрока, они рассчитываются по той же формуле, что и координаты объекта уровня. Организацию загрузчика уровня можно посмотреть в [Приложение 2](#приложение-2-реализация-загрузчика-уровней)  
  
Таким образом, разработанный загрузчик уровней на основе чтения ASCII-моделей показал высокую эффективность. Использование текстовых файлов для описания уровней обеспечивает простоту создания и гибкость модификации уровней, благодаря чему он поддерживает даже пользовательские уровни, высокую скорость загрузки и четкое разделение данных и логики.  
  
## **2.3 Архитектура проекта и разработка физики игрока и камеры**  
  
Создание физики – главная часть проекта. Для лучшей организации кода, для создания физики игрока и камеры я буду использовать классы. Они помогают объединить части кода в единую структуру, обеспечивая удобное повторное использование кода и гибкое управление сложными состояниями.  
  
### **2.3.1 Физика игрока**  
  
Для создания физики игрока, необходимо прописать базовые характеристики в конструкторе класса. В конструкторе класса были прописаны:  
  
- Координаты игрока (из функции загрузки уровня).  
  
- Размеры и смещение хитбокса.
  
- Скорость по X.  
  
- Параметры анимации: направление движения, индекс анимации, таймер анимации - для обеспечения плавности и корректности анимации  
  
- Скорость по вертикальной оси, гравитация и сила прыжка - это начальная скорость, которую получает персонаж в момент прыжка. Поскольку ось Y направлена вниз, отрицательное значение заставляет его лететь вверх. Сила прыжка равна -22, подобрано экспериментально. Если поставить значение, маленькое по модулю (например -5), то прыжок будет очень слабым, а если большое по модулю (например -50), то он будет слишком сильным.  
  
Для удобства управления в игре реализованы:

- **Койот-тайм** — возможность прыгнуть в течение нескольких кадров после падения с платформы.
- **Буфер прыжка** — запоминание нажатия прыжка, чтобы он выполнился сразу после приземления.
  
Теперь, когда базовые характеристики есть, можно приступать к разработке поведения игрока. Сначала, я реализовал функцию для прыжка, она считывает нажатия на пробел. В зависимости от возможности прыжка (см. ниже), он выполняется или нет. Чтобы не возникало двойного прыжка, также для обработки коллизии игрока с платформами, под ногами игрока создается прямоугольник. Прыжок разрешен, только если этот прямоугольник касается поверхности. Когда игрок приземляется на платформу, прямоугольник касается платформы и падение прекращается. Для корректного считывания взаимодействия игрока с объектами, объявляется функция, которая создает хитбокс игрока с мировыми координатами игрока. Для передвижения игрока была создана функция, которая обеспечивает передвижение игрока по оси X. Она считывает нажатия игрока на клавиши “A” и “D” в зависимости от нажатия клавиш передвигает игрока. Теперь, необходимо сделать систему столкновений с объектами. Текущие координаты игрока записываются в  “архив координат”, затем получаем хитбокс игрока из вышеупомянутой функции. Перебирая список стен, полученный из данных уровня, мы создаем хитбокс стены. Затем, поочередно проверяем коллизии хитбокса игрока с каждым хитбоксом стены, если игрок столкнулся со стеной, то мы сбрасываем координату X. Если во время прыжка произошло столкновение головы игрока со стеной, то прыжок заканчивается и начинается падение.  
  
Теперь необходимо правильно отобразить и анимировать игрока. Сначала вычисляются экранные координаты игрока, относительно мировых координат по формуле:

- ```X_screen = X_world - X_camera - X_width // 2```,
- ```Y_screen = Y_world - Y_camera - Y_offset```.

Где:
- `X_screen`, `Y_screen` - экранные координаты игрока,
- `X_world, `Y_world - мировые координаты игрока,
- `X_camera, `Y_camera - координаты камеры (см. подраздел [2.3.2](#232-механика-камеры)),
- `X_width` - ширина хитбокса игрока,
- `Y_offset` - смещение хитбокса игрока по оси Y (используется из-за неизвестной ошибки).

В зависимости от направления игрока, проигрывается анимация движения. Подробнее организацию класса игрока можно посмотреть в [Приложение 3](#приложение-3-реализация-класса-камеры)  
  
### **2.3.2 Механика камеры**  
  
Для создания механики камеры, необходимо прописать базовые характеристики в конструкторе класса. В нем задаются координаты камеры и длина с высотой камеры, равные длине и ширине экрана (**640x360**). Для корректного отображения объектов на экране была создана функция, которая задает координаты камеры по формуле:

- ```X_camera = X_world_player - screen_width // 2```,
- ```Y_camera = Y_world_player screen_height // 2```.

Где:
- `X_camera`, `Y_camera` - координаты камеры,
- `X_world_player`, `Y_world_player` - мировые координаты игрока,
- `screen_width`, `screen_height` - ширина и высота экрана.
<a id="camera-transform"></a>
### Также, для корректного отображения объектов была создана функция, которая переводит мировые координаты в экранные по формуле

```X_screen = X_world - X_camera```,
```Y_screen = Y_world - Y_camera```.

Где:
- `X_screen`, `Y_screen` - экранные координаты объекта,
- `X_world`, `Y_world` - мировые координаты объекта,
- `X_camera`, `Y_camera` - координаты камеры.

Итак, созданная физика является ядром проекта. Использование классов позволило добиться лучшей организации кода и гибко управлять состояниями игрока. Реализованные механики обеспечивают приятное управление и правильное отображение игрока. Подробнее организацию класса камеры можно рассмотреть в [Приложение 4](#приложение-4-реализация-класса-игрока)
